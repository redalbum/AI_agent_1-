Сделать агента stateful по знаниям о БД
Кэшировать валидные объекты/поля/связи и переиспользовать между сессиями.
Снижает «угадывание» и количество recovery-циклов.
Разделить стратегию на 2 режима: fast-path и safe-path
fast-path: минимум шагов для типовых запросов.
safe-path: обязательные проверки схемы/полей/доступов для сложных или неясных задач.
Переключение автоматически по риску запроса.
Ввести бюджет выполнения на уровне оркестрации
Лимиты не только на токены, но и на число циклов Plan→Execute→Recover.
При превышении — ранний stop с техничным, полезным summary.
Формализовать DSL как “контракт-first”
Единый валидатор + нормализатор до execute.
Строгие версии контракта (dsl_version) и совместимость между версиями.
Укрепить стратегию деградации (graceful degradation)
Если «идеальный» путь не проходит, агент должен уметь отдавать частично полезный результат, а не застревать.
Например: «вот данные по ближайшему валидному объекту + что нужно уточнить для точного ответа».
Сделать observability как у production-сервиса
Трассировка по trace_id, метрики p50/p95 latency, token cost per intent, recovery success rate.
Дашборд причин деградации качества (а не только pass/fail).
Улучшить качество промптов через policy-layer
Не раздувать один огромный промпт, а собирать его из модульных политик.
Политики: безопасность, формат ответа, SQL/1C-style ограничения, anti-loop.
Развивать автотесты как систему качества, а не только gate
Добавить regression-набор «из боевых логов».
Делать дифф-проверку не только по pass/fail, но и по эффективности (токены, шаги, retries).
Стабилизировать RAG-контур
Регулярное обновление индекса, контроль качества чанков, отсев шумных кандидатов.
Ранжирование по «практической пригодности для RunQuery», а не просто текстовой похожести.
Управление рисками изменений
Canary rollout: сначала на части запросов.
Авто-rollback промпт/политик при резком росте ошибок или стоимости.
